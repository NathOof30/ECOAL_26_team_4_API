### Contexte backend (IMPORTANT)
- Backend: **Laravel 12 + Sanctum**.
- Base API: `http://127.0.0.1:8000/api` (configurable via `.env` côté front).
- Auth protégée: middleware `auth:sanctum` avec **token Bearer** (personal access tokens).
- La colonne `users.remember_token` existe, mais l’API renvoie un **`access_token` Sanctum**: le “remember” se fait côté front en stockant le token (localStorage), **pas via cookies**.

### Headers et réseau
- Toujours: `Accept: application/json`.
- Si body JSON: `Content-Type: application/json`.
- Si route protégée: `Authorization: Bearer <access_token>`.
- Pas besoin de CSRF/cookies pour ces endpoints.
- Attention CORS: prévoir que le front tourne sur un autre port (Vite), donc l’origine doit être autorisée côté Laravel (mais ne pas gérer CORS dans ce code front).

### Règles métier / payloads principaux (rappel)
**Auth**
- Register (`POST /register`) body:
  - `{ name, email, password, avatar_url?:string|null, nationality?:string|null, user_type?: "admin"|"editor"|"user" }`
  - Réponse: `{ access_token, token_type: "Bearer", user }`
- Login (`POST /login`) body:
  - `{ email, password }`
  - Réponse: idem register.
- Logout (`POST /logout`): pas de body, invalide le token courant.
- Me (`GET /user`): renvoie l’utilisateur connecté.

**Collections**
- 1 collection max par user, sinon 403 `{ message: "User already has a collection." }`.
- `POST /collections` body: `{ title, description?:string|null }` (user_id géré backend).

**Categories**
- Table `category`, champ `title`.
- `POST /categories` body: `{ title }`.

**Criteria**
- Table `criteria`, PK = `id_criteria`.
- `POST /criteria` body: `{ name }`.

**Items**
- `POST /items` nécessite une collection existante pour le user.
- Body:
  ```json
  {
    "title": "Zippo 1941 Replica",
    "description": "...",
    "image_url": null,
    "status": false,
    "category1_id": 1,
    "category2_id": null
  }
  ```

**Scores (item_criteria)**
- `POST /item-criteria` body:
  - `{ "id_item": number, "id_criteria": number, "value": 0|1|2 }`

### Détails attendus des modules

#### 1. Client API (`api/client.js`)
Implémenter un client minimaliste:
- `getBaseUrl()` → lit `import.meta.env.VITE_API_BASE_URL`.
- `apiFetch(path, { method = "GET", body, auth = false })`:
  - Construit l’URL complète (trim des `/`).
  - Ajoute `Accept: application/json`.
  - Ajoute `Content-Type: application/json` si `body` défini.
  - Ajoute `Authorization: Bearer <token>` si `auth` true et token présent (fourni par le contexte d’auth).
  - Mesure la durée (`performance.now()`).
  - Gère réponses:
    - 204 → retourne un objet sans tenter `json()`.
    - Sinon lit `text`, essaie `JSON.parse`, mais conserve le **texte brut**.
  - Retourne un objet:
    - `{ ok, status, durationMs, json, text, error, url, method, headersSent, bodySent }`.

Prévoir une fonction utilitaire:  
`setTokenGetter(fn)` pour que le client puisse récupérer le token courant depuis le contexte.

#### 2. Contexte d’auth (`context/AuthContext.jsx`)
- Stocker:
  - `user` (ou `null`)
  - `token` (string ou `null`)
  - `loading` / `error` éventuels pour les actions.
- Persistant:
  - Lire/écrire le `token` dans `localStorage` (clé genre `"ecoal_token"`).
- Fournir un hook `useAuth()` pour accéder facilement à:
  - `login(email, password)`
  - `register({ name, email, password })`
  - `logout()`
  - `fetchMe()` (GET `/user` pour rafraîchir le profil)
- Après `login` / `register`: stocker `access_token`, le fournir au client API, et éventuellement appeler `fetchMe`.
